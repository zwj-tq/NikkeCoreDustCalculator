# 移动端适配设计

## 背景

当前项目是一个可直接部署的静态页面工具，技术形态为：

- `index.html`
- `styles.css`
- `app.js`
- `nikke-data.js`

页面已经具备桌面端较高的信息密度，适合在宽屏下一次性查看参数、活动、额外来源、策略、图表和每日明细。但在移动端存在以下问题：

- 页面信息层级仍然偏桌面优先，首屏承载内容过多
- 多列表单编辑器在小屏上可读性和可编辑性较差
- 宽表格在手机上难以快速浏览
- 顶部操作区和状态区在触屏设备上的优先级不够清晰
- 当前响应式规则主要是压缩列数，还没有针对移动端交互形态做重新设计

本次改造目标不是引入新的前端框架，而是在保持静态站架构的前提下，完成一次轻中度重构，使页面在手机上真正可用，并继续保留桌面端高密度工作台体验。

## 目标

本次移动端适配的目标如下：

- 保留原生静态页面架构，不引入 Tailwind、React、Vue 等新框架
- 保留现有模拟计算逻辑和数据结构，避免影响核心算法
- 将页面从“桌面优先压缩”升级为“移动端与桌面端双形态展示”
- 让高频操作和核心信息在手机上更容易触达
- 让多条记录编辑在手机上具备清晰、稳定、可点击的卡片式交互
- 让结果区在手机上优先展示结论，而不是先暴露完整明细表格

## 非目标

本次不包含以下内容：

- 不引入构建工具链
- 不迁移到组件化框架
- 不重写核心模拟算法
- 不重做视觉品牌或整体风格方向
- 不新增复杂的后端、存储或账号能力

## 设计原则

### 1. 保持静态站方案

继续使用当前的静态网页模式，保证 GitHub Pages 部署方式不变，不新增本地运行依赖。

### 2. 展示层重构优先于计算层重构

本次改造重点放在页面信息架构、DOM 输出结构和 CSS 规则上。计算逻辑继续复用现有状态与结果模型。

### 3. 移动端不是桌面端缩小版

对手机视图，不强求复刻桌面布局，而是针对触屏和窄屏重新组织内容顺序与交互形式。

### 4. 桌面和移动共享数据，不共享所有 DOM 结构

桌面端和移动端应继续使用同一份状态，但在关键区域允许渲染不同的 DOM 结构，以降低小屏适配成本。

### 5. 优先解决最影响体验的区域

优先处理以下问题最明显的模块：

- 顶部工具栏和状态区
- `events / extras / strategies` 编辑区
- 策略汇总
- 每日明细

## 页面信息架构

移动端页面按以下层级组织：

1. 顶部操作区
2. 配置区
3. 结果区
4. 详情区

### 顶部操作区

顶部操作区保留核心主操作：

- `重新计算`

次级操作降级处理：

- `导出当前明细 CSV`
- `导出图表 PNG`

在移动端，这两个导出按钮不再与主按钮并列铺开，而应收进“更多操作”区域，避免首屏操作过载。

状态展示在移动端缩减为少量高价值信息，例如：

- 当前最佳策略
- 最终等级
- 策略数

桌面端仍可保留当前较完整的状态卡布局。

### 配置区

配置区包含以下模块：

- 基础参数
- 主线更新
- 活动
- 额外来源
- 策略

移动端中，这些模块统一表现为可折叠卡片区块。

默认展开：

- 基础参数
- 策略

默认折叠：

- 主线更新
- 活动
- 额外来源

这样可以让用户优先完成关键输入，减少初始认知负担。

### 结果区

结果区包含：

- 核心指标卡
- 主图表
- 策略汇总

移动端优先展示结论，图表放在指标卡之后，汇总放在图表之后。

### 详情区

详情区仅包含：

- 每日明细

移动端默认折叠该区块，只有用户明确需要时才展开查看。

## 桌面端与移动端的双形态策略

### 保持单一数据源

以下内容继续共享当前状态和结果数据：

- 参数状态
- 活动状态
- 额外来源状态
- 策略状态
- 模拟结果
- 明细结果

### 允许双形态渲染的区域

以下区域在桌面端和移动端允许输出不同 DOM：

- 工具栏与状态区
- 多条记录编辑器
- 策略汇总
- 每日明细

### 共享渲染逻辑但切分输出函数

JS 中将保留统一的数据准备流程，但分离桌面端和移动端的展示函数，减少单个函数中堆叠大量条件判断。

## 编辑区设计

### 基础参数

基础参数继续使用表单结构，但在移动端改成单列布局，按逻辑分组展示，保持现有分组标题与说明。

不再追求在手机上并排展示多个字段。

### 主线更新

主线更新区继续以图表为主要入口。

移动端策略如下：

- 保留主线图表
- 点击图表节点时继续使用弹层编辑
- 如有必要，可在图表下方补充简要记录列表，但不强制引入复杂网格编辑器

这样可以保留当前交互优势，避免将复杂编辑结构硬塞进窄屏。

### 活动、额外来源、策略

这三个区块改为“桌面保留网格、移动端改为卡片”。

移动端卡片规则：

- 每条记录一张卡片
- 卡片顶部展示记录名称、类型或状态、删除按钮
- 核心字段纵向排列
- 次要字段收进“更多设置”或卡片下半区
- 新增按钮固定在区块头部

推荐优先展示的字段如下。

活动：

- 名称
- 开始日期
- 持续天数
- 奖励数量

额外来源：

- 名称
- 频率
- 数量
- 起始范围

策略：

- 名称
- 类型
- 目标日或目标等级
- 启用状态

备注等辅助字段可放在卡片次级区域。

## 结果区设计

### 核心指标卡

移动端指标卡改成单列或双列小卡，不强行保持桌面端横向 4 列或 5 列布局。

指标卡优先展示最重要的结果字段，避免信息噪音。

### 图表区

移动端继续保留主要折线图。

策略如下：

- 降低图表高度，避免首屏被图表完全占据
- 保留 tooltip 读取能力
- 不新增重型交互控件

柱状图可保留，但在移动端中应放在折线图之后，并视空间情况弱化默认优先级。

### 策略汇总

桌面端继续使用表格。

移动端改为策略摘要卡列表，而不是保留一张宽表强制横向滚动。每张摘要卡展示：

- 策略名
- 类型
- 最终等级
- 剩余箱子
- 累计开箱

卡片支持点击后切换每日明细策略。

## 每日明细设计

### 桌面端

继续保留当前完整明细表格，维持适合大屏分析的形式。

### 移动端

默认展示按天卡片列表，并提供“查看完整表格”切换入口。

移动端每张明细卡优先展示：

- 天数
- 等级
- 级内进度
- 箱子
- 当日开箱
- 日常芯尘
- 额外芯尘

说明文字置于卡片底部。

如果用户需要完整字段对照，可切换到横向滚动表格视图。

这样可以兼顾：

- 手机上的快速浏览
- 对完整数据列的保留需求

## CSS 结构调整

`styles.css` 将按照以下层次整理：

### 1. Tokens

整理现有变量，补充移动端需要的尺寸和断点变量，例如：

- 间距
- 卡片内边距
- 触控最小点击尺寸
- 工具栏安全留白
- 断点值

### 2. Layout

布局层负责：

- 页面宽度
- 栅格列数
- 区块间距
- 顶部工具栏布局
- 配置区和结果区排列

### 3. Components

组件层负责：

- 按钮
- 字段
- 状态卡
- 区块卡
- 移动端编辑卡
- 汇总卡
- 明细卡

### 4. Responsive Overrides

媒体查询只做布局切换与组件级差异覆写，不堆叠零散补丁。

## 断点策略

建议保留并明确以下 3 档：

- `>= 1080px`：桌面高密度
- `721px - 1079px`：平板/小屏笔记本，中密度双列
- `<= 720px`：手机单列、卡片化

重点不是所有元素都缩放，而是让不同断点采用不同展示模式。

## JS 重构方案

当前 `app.js` 的主要问题不是算法复杂，而是渲染逻辑集中，后续移动端改造容易在单个函数里堆分支。

建议新增渲染边界：

- `isMobileLayout()`
- `renderToolbar()`
- `renderMetrics()`
- `renderParams()`
- `renderMainlinesSection()`
- `renderCollectionSection(type)`
- `renderSummaryDesktop()`
- `renderSummaryMobile()`
- `renderDetailDesktop()`
- `renderDetailMobile()`

其中：

- 计算逻辑保持共用
- 渲染入口保持统一
- 桌面与移动的 DOM 输出分别封装

### 渲染切换原则

在以下情况下按布局模式切换渲染：

- 初始化时
- 窗口尺寸变化时
- 数据变化后重新渲染时

切换时要确保：

- 当前选择的明细策略不丢失
- 图表实例在布局变化后继续正常 resize
- 折叠状态尽量保持一致

## 错误处理与兼容性

### 图表与尺寸变化

移动端在展开、折叠和视口尺寸切换时，需要继续调用图表 resize，避免图表容器尺寸错乱。

### 长文本和说明字段

移动端卡片中需允许说明文字换行，不再沿用桌面表格的 `nowrap` 策略。

### 表单触控区域

按钮、删除图标、卡片点击区域需要满足触控尺寸要求，降低误触。

### 默认折叠状态

如果某一配置区包含错误输入或用户刚新增记录，应考虑自动展开相关区块，避免内容被折叠后难以发现。

## 测试策略

本次改造需要至少覆盖以下验证点：

### 布局验证

- 桌面端布局未明显退化
- 平板端列数和卡片布局正确
- 手机端单列布局完整

### 交互验证

- 折叠区块可正常展开和收起
- 新增、编辑、删除活动/额外来源/策略在手机上可顺畅完成
- 点击策略汇总可切换明细策略
- 每日明细卡片和完整表格视图切换正常

### 数据验证

- 重构后模拟结果不变
- 导出 CSV、导出 PNG 功能继续正常
- 图表在窗口变化后继续正常显示

## 分阶段实施顺序

建议按以下顺序实施：

1. 重构顶部工具栏和状态区
2. 重构 `events / extras / strategies` 编辑区
3. 重构策略汇总与每日明细双形态展示
4. 整理基础参数区与主线更新区的小屏表现
5. 收尾全局断点、间距、表单触控体验

这样可以优先拿到移动端体验最明显的收益，同时降低一次性改动范围。

## 决策结论

本项目的移动端适配采用以下方案：

- 保留原生静态站技术栈
- 不引入 Tailwind 或组件化框架
- 使用桌面端与移动端双形态展示
- 共享同一份数据和计算逻辑
- 优先将记录编辑区、策略汇总区、每日明细区卡片化

该方案兼顾：

- 当前项目规模
- GitHub Pages 静态部署方式
- 移动端真实可用性
- 后续维护成本
